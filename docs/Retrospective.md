# 회고 (Retrospective)

상태: 초안. 프로젝트 마무리 단계에서 다시 손본다. 지금은 현재 구조가 어떤 가정 위에
서 있는지, 어떤 모델을 골랐고 무엇을 적용해 보려는지, 당장 포기했지만 나중에 해 보면
좋을 것은 무엇인지, 그리고 analyze/verify/evaluate 세 검증 명령을 어떻게 짰는지를
정리해 둔다.

확정된 계약은 `specs/`에, 배경 설명은 `docs/`에 있다. 이 문서는 그 결정들의 회고지
계약이 아니다. 숫자나 인터페이스가 어긋나면 `specs/`가 정본이다.

## 1. 현재 구조의 가정과 가설

지금 아키텍처는 몇 가지 베팅 위에 서 있다. 맞다고 증명된 게 아니라, 맞다고 보고
구조를 그쪽으로 기울인 가설이다.

- **스타일은 하드코딩이 아니라 측정 대상이다.** 레퍼런스에서 컷 리듬, 오디오
  다이내믹, 팔레트, 톤을 뽑아 `VideoProfile`로 만들고, 생성은 그 데이터를 씨앗으로
  돈다. 빠른 컷 레퍼런스와 느린 컷 레퍼런스가 서로 다른 스토리보드 리듬을 만들어야
  한다는 게 핵심 가설이다. 이게 맞아야 "같은 엔진, 다른 입력, 다른 결과"가 성립한다.
- **분석기가 곧 채점기다(단일 잣대).** 레퍼런스를 프로파일링하는 코드가 생성물도
  프로파일링한다. 그래서 레퍼런스와 생성물이 같은 자로 재진다. 잘 만든 레퍼런스가
  검증을 통과한다는 전제를 회귀 기준선으로 쓴다.
- **스키마가 유일한 경계다.** 분석과 생성은 pydantic 스키마로만 통신한다. 가장 바뀌기
  쉬운 부분(이미지, 영상 백엔드)을 스키마 뒤에 가두면 백엔드 교체가 한 단계만 건드린다는
  가정이다. 이 경계가 흐려지면 설계를 다시 본다.
- **결정론 층이 우선이고, 모델은 그 위에 얹힌다.** 컷, 오디오, 팔레트 같은 수치는
  로컬 라이브러리가 재현 가능하게 내고, 모델은 지각적 묘사만 더한다. 결정론 수치를
  모델이 덮어쓰지 않는다. 이래야 분석이 키 없이도, 모델이 흔들려도 안정적으로 돈다.
- **로컬 단독 실행이면 충분하다.** 노트북에서 도는 CLI 하나로 끝낸다. ffmpeg 영상
  파이프라인이 서버리스 제약(타임아웃, 파일시스템)에 맞지 않으니 웹 프런트와 서버리스
  배포는 범위 밖이다. 코어는 CLI가 인프로세스로 부르는 패키지라, 나중에 웹 계층이 같은
  코어를 감쌀 수 있게만 열어 둔다.
- **단일 키로 진입 장벽을 낮춘다.** 필수 키는 `GEMINI_API_KEY` 하나로, 분석과 이미지
  생성을 덮는다. GCP 자격이 있으면 Vertex lane을 우선해 크레딧을 쓰되, 단일 키는
  폴백으로 항상 보장한다. 영상, 음악, 보이스 모델은 옵션이고 쓸 때만 키를 더한다.
- **voice는 나레이션을 기본으로 한다.** 실측 결과, voice를 먼저 만들어 영상 모델에
  주입해 립싱크시키는 방식은 무리였고, 화면 인물이 직접 말하는 on-camera 립싱크는 컷이
  나뉘면 컷마다 목소리 톤이 흔들렸다. 그래서 기본은 화면과 분리된 나레이션(ElevenLabs
  또는 Google TTS Chirp 3)으로 캐릭터 음색을 길게 연속으로 뽑는다. 화면 인물이 직접
  말하는 연출은 꼭 필요할 때만, 그것도 멀티컷에서 목소리 일관이 필요하면 Kling O3 Pro
  reference-to-video로만 간다. 영상 모델 예산이 없어도 스틸을 켄 번스 모션으로 같은
  조립 경로에 태워 끝까지 돈다는 가정으로 저비용 폴백을 깐다.
- **사람 확인 게이트를 일반화할 수 있다.** 모든 중요한 단계 뒤에 같은 게이트 추상이
  붙고, ask(챗)/pass(`--force-step-pass`)/런 모드(전부 통과)가 한 그래프 위에서 돈다는
  가정이다.
- **결정은 코드가 아니라 노드별 LLM이 문맥으로 내린다.** 스타일 선택(캐릭터 외모, 음악
  장르·강도, 후크 유형, 촬영 환경, 대사, 스토리보드 컷·카메라, 제품 스펙)을 코드나 프롬프트에
  하드코딩하지 않고, 각 노드가 LLM으로 입력·레퍼런스·주인공 문맥에 맞게 정한다. 기본값은
  '편향'일 뿐, 입력이 분명히 다른 걸 원하면 그걸 존중한다(예: 매력적 미인이 기본이되 평범한
  인물을 요청하면 따른다). 캐릭터를 모든 노드가 공유 문맥으로 써서 "이 캐릭터라면?"으로
  대사·톤·페이싱을 정한다.
- **plan/execute를 LangGraph로 표현한다(2026-07-01 전환).** 초안은 순차 함수였지만 실제
  `StateGraph`로 이관했다([workflows.md]). plan은 조건부 엣지로 **hook ↔ storyboard 핑퐁**을
  돈다(나중 추가): storyboard 노드가 주어진 훅을 전체 스토리에 녹여 보고 잘 안 맞으면 개선
  힌트와 함께 hook 노드를 재호출해 다시 짠다(최대 재시도 상한). "훅과 스토리보드는 따로가
  아니라 함께 맞물려야 한다"는 관찰을 그래프 구조로 옮긴 것이다. 노드 span은 로컬 trace(항상)
  +Langfuse(옵션)로 남는다. 창의 의도는 plan이 담고 execute는 그 plan을 충실히 렌더한다
  (execute가 하드코딩 룰로 비슷한 결과에 수렴하지 않게).

## 2. 모델 선정과 적용 의도

용도마다 무엇을 1차로 두고 무엇을 폴백/홀딩으로 뒀는지의 요지다. 선택의 정본과 근거는
[../specs/ai-model-records.md](../specs/ai-model-records.md)에 있다. 여기서는 "왜 그렇게
골랐고 무엇을 실제로 적용해 보려는가"만 적는다.

- **레퍼런스 비전 분석**: Gemini Flash 계열을 1차로. 빠르고 싸고 멀티모달이라 지각
  묘사용으로 충분하다. 깊이가 모자라면 Pro로 한 단계 올린다. 분석 층은 결정론 수치가
  더 중요해 굳이 키를 늘리지 않는다(Claude 비전은 홀딩).
- **기획·카피 LLM(컨셉, 훅, 스토리보드, 대사, 톤)**: 여기만 예외로 Gemini 3.1 Pro와
  Claude Opus를 일급으로 나란히 비교한다. 두 키가 이미 있어 추가 비용 없이 같은 작업을
  둘에 돌려 카피와 내러티브 품질을 본다. 잠정 1차는 Gemini 3.1 Pro로 두되, 작업별로
  나은 쪽을 채택해 굳혀 갈 계획이다.
- **이미지 생성(에셋 시트, 패널 스틸)**: Google Nano Banana 계열(Gemini 네이티브
  이미지) 단일 경로. 단일 키 안에서 돌고 한 이미지에 멀티뷰를 렌더링해 에셋 시트에 맞다.
  이미지는 외부 백엔드 없이 Nano Banana 하나로 좁혔다.
- **image-to-video**: 개발 검증 기본은 Veo 3.1 Fast로 못박았고, 항상 Vertex AI lane으로만
  호출한다. Fast가 start image 기반 뷰티 사용 컷에서 가장 무난했다. Standard/Pro급 Veo는
  중요한 히어로 컷의 고품질 승격 옵션으로만 둔다. Lite는 start/end control board에서 split
  artifact가 생겨 기본 후보에서 내린다. 다만 성능상 최선의 선택은 Kling O3다. 특히 O3 Pro
  reference-to-video는 캐릭터 설정 이미지, 제품 이미지, start/action image, 멀티샷 storyboard를
  함께 넣었을 때 캐릭터·제품·액션 일관성이 가장 좋았다. 지금은 전체 개발 cycle을 Veo Fast로
  마무리하고, 사용자가 "Kling으로 가자"고 지시하면 그때 기본 영상 모델을 Kling O3로 바꾼다.
- **BGM**: Lyria 생성을 1차로, 사용자 제공 파일이나 무음을 폴백으로.
- **voice(나레이션 기본)**: ElevenLabs를 선호, 또는 Google TTS Chirp 3로 캐릭터 음색
  나레이션을 길게 뽑는다. on-camera 발화는 역동적 컷에서만, 멀티컷 일관이 필요하면 Kling
  O3 Pro로. voice 주입 립싱크와 전용 아바타는 제외.

## 3. 당장 포기했지만 아쉬운 것 (추후 적용 후보)

시간과 범위 때문에 지금은 접었지만, 여유가 생기면 붙여 보면 좋을 것들이다.

- **전용 토킹헤드 아바타 파이프라인.** voice 자체는 되도록 쓰는 쪽으로 바꿨다(캐릭터
  매칭 voiceover, 영상 모델 네이티브 발화 on_camera). 다만 HeyGen류 전용 립싱크 아바타
  파이프라인은 구현 부담과 깨짐 때문에 만들지 않는다. 영상 모델 네이티브 발화로 충분치
  않은 정교한 립싱크가 필요해지면 그때 별도 경로를 볼 만하다.
- **기획·카피 LLM 자동 라우팅.** 지금은 Gemini 3.1 Pro와 Claude Opus를 명시 우선순위로
  비교만 한다. 작업 유형별로 더 나은 모델을 자동으로 고르는 라우팅은 현 규모에 과해
  접었지만, 작업 패턴이 쌓이면 가치가 있다.
- **Seedance 검열 리스크.** Seedance 2.0 Fast는 멀티샷과 영화적 연출 잠재력은 있지만,
  storyboard + persona character fit + product reference를 함께 넣은 테스트에서 얼굴/인물
  reference가 likeness/private information 가능성으로 provider validation에 막혔다. 이
  프로젝트는 캐릭터 외모와 제품을 같이 고정해야 하므로, Seedance는 품질 문제가 아니라
  사용성 자체가 불안정한 후보로 본다.
- **Kling O3가 성능상 최선.** 현시점 이 프로젝트의 영상 생성 품질만 보면 Kling O3가 가장
  강하다. 최선 경로는 O3 Pro reference-to-video다. 적절한 prompt와 함께 start/action image,
  구체적인 캐릭터 설정 이미지, 구체적인 제품 이미지, 멀티샷 storyboard를 reference로 넣으면
  복잡한 뷰티 제품 컷에서도 일관성이 좋다. 컷별로 잘라 생성하고 조립하는 방식이 더 고품질이지만,
  멀티샷이 필요하면 O3 `multi_prompt`로 한 번에 밀어붙일 수 있다.
- **Kling O3 image-to-video는 Veo 치환이 쉽다.** start image와 end image를 넣을 수 있어
  Veo 3.1 image-to-video와 비슷한 형태로 바꿔 끼우기 좋다. 마스크팩 테스트에서는 start/end
  이미지를 먼저 만든 뒤 O3 image-to-video에 넣는 방식이 reference-to-video보다 액션 전환을 더
  명확하게 만들었다. 그래도 캐릭터·제품·스토리보드 reference를 많이 묶는 복잡한 컷은
  reference-to-video가 더 맞다.
- **Veo start image 컷 제어.** Veo는 텍스트만으로는 제품, 캐릭터, 시작 동작, 끝 상태가
  안정적으로 고정되지 않았다. Nano Banana로 컷별 start image를 먼저 만들고, 필요하면 한 장
  안에 start/end keyframe board를 만들어 프롬프트로 "split-screen이 아닌 연속 full-screen
  shot"이라고 강하게 설명하는 경로가 가장 잘 먹혔다. 이 방식은 Veo Fast/Standard에서 좋았고,
  Lite는 split artifact가 생겼다.
- **상위 이미지 모델 승격.** 1차로 충분하다고 보고 더 상위 이미지 모델은 홀딩했다.
  스틸 품질이 영상 품질로 전파되므로, 룩이 결과를 좌우하는 컨셉에서 승격 효과를
  재 볼 만하다.
- **whisperX 강제 정렬.** 자막 타이밍을 스토리보드에서 가져오는 주 경로에선 불필요해
  미뤘다. 보이스오버를 본격적으로 켜면 음성에서 타이밍을 따오는 정렬이 필요해진다.
- **외부 스톡 음악/소스 보강.** 기본 경로에서 스톡 검색 provider를 뺐다. 생성 BGM과
  제공 파일 두 경로로 충분하다고 봤지만, 저비용 합성 경로를 더 두텁게 하려면 다시 볼 수
  있다.
- **관측 기반 튜닝 루프.** 로컬 trace와 Langfuse 배선은 설계했으나
  ([logging-strategy.md](../specs/logging-strategy.md)), 트레이스를 모아 게이트 점수와
  비용을 보고 자동으로 개선하는 루프까지는 못 갔다.

## 4. analyze, verify, evaluate를 어떻게 짰나

세 명령은 "분석기가 곧 채점기"라는 한 원리에서 갈라져 나온 단일 잣대다. analyze가
프로파일을 만들고, verify와 evaluate가 각각 하드/소프트로 판정한다. 셋 다 레퍼런스와
생성물에 같은 코드를 댄다. 그래프가 아직 미구현이어도 셋은 단독 CLI로 지금 동작한다.

### analyze (프로파일링)

- **하는 일**: 영상(로컬 파일 또는 URL) -> `VideoProfile`(JSON). URL이면 먼저
  내려받은 뒤 분석한다.
- **구성**: 두 계층이다. 결정론 로컬 층(ffprobe, PySceneDetect, librosa, OpenCV)이
  컷 분포, 오디오 다이내믹, 팔레트, 밝기를 재현 가능하게 내고, Gemini 층이 톤, 자막
  스타일, 후크 같은 지각 필드를 더한다. 결정론 수치는 지각 층이 덮어쓰지 않는다.
- **플래그**: `--no-gemini`면 결정론 수치만 내 키 없이도 돈다. `--out`으로 JSON을
  저장(미지정 시 stdout). `--cookies-from-browser`로 로그인 필요 URL을 받는다.
- **연계**: `add-reference`는 URL 하나로 다운로드 -> analyze + evaluate -> 저장 ->
  카탈로그까지 묶는다. 레퍼런스를 들일 때 프로파일과 Rubric 평가를 함께 남긴다.

### verify (Conformance, 하드 pass/fail)

- **하는 일**: 영상이 의도한 템플릿대로 기술적으로 온전히 만들어졌고 노드 산출물이
  빠짐없이 머지됐는지 검증한다. 계약은
  [../specs/conformance-gate.md](../specs/conformance-gate.md).
- **구성**: 결정론 체크(미디어 무결성, 템플릿 적합성, 볼륨 LUFS/클리핑, 머지 무결성,
  스키마 검증)에 더해, 선택적 VLM 지각 체크(자막 위치, 효과, 전환)가 붙는다. VLM은 외부
  호출이라 키가 없으면 skip이고 skip은 통과로 친다.
- **입력**: 영상에 더해 `--input`(generation_input), `--storyboard`, `--manifest`(run)를
  선택으로 받는다. 레퍼런스처럼 템플릿/매니페스트가 없으면 intrinsic 체크만 돌고 나머지는
  skip한다. 잘 만든 레퍼런스는 모두 PASS여야 한다(회귀 기준선).
- **출력**: `ConformanceReport`(JSON). fail이 하나라도 있으면 exit code가 0이 아니다.
  그래프에서는 fail 시 rubric으로 가지 않고 결함 카테고리로 해당 샷 재생성을 트리거한다.
  `--no-vlm`이면 결정론 체크만 돈다.

### evaluate (Rubric, 소프트 0~100)

- **하는 일**: conformance를 통과한 영상이 콘텐츠로서 먹히는지를 드라이버 Rubric으로
  채점한다. 계약과 수식은 [../specs/rubric.md](../specs/rubric.md).
- **구성**: D1~D7 차원을 채점하되, D1과 D2가 곱셈 게이트로 작동하고 D3~D7이 가산된다.
  `gated_score`(게이트 반영)와 `flat_score`(단순 합)를 함께 낸다. 채점은 Gemini 저지가
  맡는다.
- **플래그**: `--no-gemini`면 저지를 건너뛰고 채점 불가(`scored=false`)로 보고한다.
  `--out`으로 JSON 저장.
- **출력**: `RubricResult`(JSON)와 한 줄 요약(gated/flat 점수, 게이트 통과 여부, 통과/미달).

### 셋의 관계

순서가 있다. analyze가 잣대(프로파일)를 만들고, verify가 먼저 하드로 거른 뒤, 통과한
것만 evaluate가 소프트로 점수 낸다. 결과 산출물은 분석은 `profiles/`, 검증과 평가는
`evals/`(conformance는 `evals/conformance/`)에 남기며, 모두 gitignore라 재생성 가능한
산출물로 둔다.
