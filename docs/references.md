# 선행 사례: 숏폼 광고 생성 도구

이 문서는 경쟁 제품(SaaS)에서 배울 생성 흐름과 제품 패턴을 다룬다.

기존의 AI 숏폼/UGC 제품 광고 생성 도구들을 살펴보고, 이 프로젝트에 가져올 만한 패턴을
추렸다. 결론부터 말하면, 이 분야는 [pipeline-design.md](pipeline-design.md)의
파이프라인과 거의 똑같은 단계형 흐름으로 수렴했고, 대부분의 도구는 앞쪽 준비 단계는
강하지만 마지막 영상 단계에서 약하다. 바로 거기에 기회가 있다. 차별화는 클립 화질이
아니라, 끝까지 실제로 돌아가는 시스템이다.

예외가 하나 있다. 현시점에서 가장 앞선 도구는 영상 단계까지 완주하면서, 단순 단계형
흐름을 넘어 **컨셉 템플릿 라이브러리, 샷 단위 콘티 에이전트, 제품 affordance 주입,
생성물 품질 검사 후 샷 단위 재생성**까지 갖췄다. 아래 "콘티 에이전트로 한 단계 더"는 그
도구에서 배운, 우리 파이프라인을 끌어올릴 패턴들이다.

## 공통 흐름

도구마다 생성은 같은 방식으로 분해된다.

```
제품 (URL 또는 이미지) + 소수 옵션
   ▼
캐릭터 / 모델  (고정된 페르소나)
   ▼
키 히어로 스틸 (룩 레퍼런스)
   ▼
스토리보드     (비트별 패널, 캐릭터 + 제품 고정)
   ▼
보이스 / 자막  (비트별 내레이션)
   ▼
영상 조립
```

우리의 `asset bible -> storyboard -> subtitles/voice -> video`와 같은 분해다. 매핑이
정확히 맞아떨어지므로, 이 단계형 설계는 추측이 아니라 실제로 출시된 도구들이 만들어지는
방식이다.

## 살펴본 도구

| 도구 | 잘하는 것 | 부족한 것 |
|---|---|---|
| Creatify | 에이전트식 단계별 흐름, 높은 스틸 프레임 퀄리티, ROAS 추적 | 영상 출력은 유료 |
| Arcads | AI 배우 UGC 광고, 배우별 재사용 "워크플로", 강한 제품 완성도 | 유료 전용 |
| UGCFast | 제품 URL 하나에서 페르소나 매칭 변형, 실행당 다수 변형 | 테스트 중 생성 불안정 |
| Sandpit AI | 브랜드 분석 -> 제품 카탈로그, 명시적 캠페인 목표 택소노미 | 이미지 위주, 영상 약함/없음 |
| HeyGen | 토킹헤드 아바타, 탄탄한 립싱크 | 토킹헤드 외에는 성의 없음 |
| invideo AI | 완전히 노출된 단계형 흐름(캐릭터 -> 히어로 스틸 -> 스토리보드 -> TTS -> 영상), image-to-video 모델 선택 | (우리 흐름에 가장 가까운 레퍼런스) |
| The Influencer AI | 트레잇 기반 페르소나 빌더(성별, 나이, 스킨톤, 헤어, 체형), 프리뷰 후 고정 | 평범한 출력 퀄리티 |
| Higgsfield | 영상까지 깔끔하게 완성, 컨셉 템플릿 라이브러리, 일관된 캐릭터, 제품 사용 장면의 구체성 | (현시점 가장 강한 레퍼런스, 영상 단계까지 완주) |

## 우리가 채택할 패턴

**1. 페르소나/제품 고정을 모든 패널에 물려준다.** invideo의 스토리보드 프롬프트는
`LOCKED PROTAGONIST`와 `LOCKED PRODUCT`를 명시하고, 모든 패널에서 이를 반복해 캐릭터,
의상, 제품, 색 그레이드를 일관되게 유지한다. 이게 바로 `StoryboardPanel`의
`subject_lock`, `product_lock` 필드의 용도다. 캐릭터와 제품 레퍼런스 이미지를 모든 패널
스틸에 넘기는 방식(pipeline-design의 "이미지, 영상 백엔드" 참고)을 뒷받침한다.

**2. 내러티브 아크를 위한 비트 구조.** 반복되는 6비트 스토리보드: 문제 -> 발견 ->
사용(히어로) -> 반응 -> 증거 -> CTA, 비트별 대략적 타이밍과 함께. Sandpit은 같은 맥락의
캠페인 목표 택소노미를 노출한다(problem→solution, before/after, social proof,
new launch, promo, informative). 비트를 하드코딩하기보다 `narrative_arc`의 프리셋으로
삼기 좋다.

**3. 단일 제품 입력, 소수 옵션.** 도구들은 제품 URL이나 이미지 하나에서 출발하고, 작은
옵션 세트(언어, 보이스 타입, 배경 음악, 톤/스타일, 자막 on/off)만 노출한다. 생성 입력
표면(`GenerationInput.meta`, `voice`, `music`, `subtitle`, `style`)에 좋은 모양이다.

**4. 완벽한 한 편보다 변형 여러 개.** UGCFast의 가치는 흠 없는 광고 하나가 아니라 한
입력에서 나오는 페르소나 매칭 변형 다수다. 스키마 기반 파이프라인이 주려는 "같은 엔진,
다른 입력 -> 다른 출력" 속성과 맞는다.

**5. 보이스오버용 감정 태그 TTS.** invideo는 text-to-speech에 인라인 감정 태그를 먹인다.
예: `[curious] ... [cheerfully] ... [excited] ...`. 선택 보이스오버 경로(pipeline-design의
"보이스오버" 참고)의 입력 포맷으로 좋고, 전달이 비트에 맞도록 잡아준다.

**6. 스틸 퀄리티가 영상 체감을 좌우한다.** Creatify는 스틸이 가장 강한 출력이다.
에셋 바이블과 패널 스틸 단계에서 이미지 품질을 우선하라는 근거다. 스틸 품질이
image-to-video 단계로 전파된다.

## 콘티 에이전트로 한 단계 더

영상 단계까지 완주하는 가장 앞선 도구는 스토리보드를 한 차원 더 끌어올린다. 다음은 우리
스토리보드와 게이트 단계에 흡수할 패턴이다.

**7. 컨셉/포맷 템플릿 라이브러리.** `narrative_arc`를 자유 텍스트가 아니라 이름 있는
프리셋 세트로 둔다(UGC, before/after, 튜토리얼, 언박싱, 셀피 후기, 직접 카메라 설명,
오버사이즈 제품 모먼트 등). 템플릿마다 샷 구성 규칙을 갖고, 일부는 특수 규칙을 가진다.
예를 들어 오버사이즈 모먼트 템플릿은 적어도 한 샷에서 제품을 비현실적으로 크게 잡는다.

**8. 컨셉 + 캐릭터 + 제품, 세 축 입력.** 생성을 컨셉 템플릿, 아바타(캐릭터), 제품 세
입력으로 받는다. 우리 `narrative_arc` + `CharacterProfile` + `ProductProfile` 조합과
정확히 대응하므로, 입력 표면을 이 세 축으로 정리한다.

**9. 샷 단위 콘티 에이전트.** 스토리보드 단계를 "콘티 작가" 노드로 명시한다. 서사를
3에서 8개 샷으로 쪼개고, 샷마다 카메라 앵글과 렌즈 느낌, 인물 표정과 제스처, 제품 위치와
크기, 배경과 로케이션, 자막을 자연어로 구체적으로 채운다. 이는 `StoryboardPanel` 필드를
실제로 어떻게 채울지에 대한 설계 지침이다.

**10. 공통 프롬프트와 샷별 프롬프트 분리.** 캐릭터, 제품, 컬러, 무드를 담는 공통
프롬프트(`global_prompt`)와 동작, 카메라, 표정을 담는 샷별 프롬프트(`local_prompt`)를
나눈다. 공통부가 모든 샷에 흐르므로 캐릭터와 제품 일관성을 잡기 쉽다.

**11. 제품 affordance 추출과 주입.** 제품 이미지나 설명에서 가능한 행동을 먼저 뽑는다
(컵은 들고 마신다, 스프레이는 흔들고 뿌린다, 가방은 메고 걷는다, 넣고 꺼낸다). 이
affordance를 샷 디스크립션에 강제로 넣으면 "그럴듯한 사용 장면"이 자연스럽게 나온다.
스토리보드 앞에 둘 작은 단계로, 사용 장면의 설득력을 크게 올린다.

**12. 유사도 채점을 넘는 품질/안전 게이트.** 생성된 클립을 다시 멀티모달 모델로 검사해
제품이 제대로 나왔는지(색, 형태, 브랜드 훼손 여부), 캐릭터가 프로필과 맞는지, 위험 요소
(민감 표현, 저작권, 로고 문제)가 없는지 본다. 문제가 있으면 해당 샷만 재생성하고
프롬프트를 자동 보정한다(로고 노출 축소, 자막 위치 조정 등). 우리 게이트가
`style_profile` 유사도만 보던 것에서 한 단계 발전한 형태다.

**13. 모델 클립 길이에 맞춘 멀티샷 분할.** 영상 모델은 보통 한 번에 짧은 길이(예: 약
15초)만 만든다. 목표 길이가 길면 샷별로 쪼개 생성한 뒤 ffmpeg로 이어 붙이고 트랜지션을
넣는다. 모델과 콘티에 따라 한 번에 멀티샷을 호출할지, 샷별로 짧게 뽑아 concat할지
전략을 고른다. 클립 길이 한계는 `meta`에서 받아 분할을 결정한다.

## 약점이 알려주는 것

Sandpit은 제대로 된 영상 단계가 없고, UGCFast는 생성 중 에러가 났으며, HeyGen은
토킹헤드에만 투자한다. 어려운 부분은 끝까지 안정적으로 돌아가는 파이프라인이다. 그래서
첫 마일스톤이 워킹 스켈레톤이다. 거칠지만 완성된 클립을 끝에서 끝까지, 영상 모델 예산
없이도 끝나도록 저비용 스틸 모션 폴백과 함께 돌린다(pipeline-design의 "첫 구현 범위"와
"저비용 폴백" 참고).
