# 회고 (Retrospective)

이 문서는 지금 구조가 어떤 가정 위에 서 있는지, 무엇이 막혔는지, 당장 포기했지만 나중에
해 보면 좋을 것은 무엇인지 정리한다. 모델 개요와 사용법은 README에, 도구 선택 근거는
[docs/ToolnModels.md](docs/ToolnModels.md)와 [../specs/ai-model-records.md](specs/ai-model-records.md)에
있으니 여기서는 겹치는 설명을 줄이고 회고에 고유한 판단만 남긴다. 숫자나 인터페이스가
어긋나면 `specs/`가 정본이다.

## 1. 핵심 한계: 제품 표현 충실도 (중요)

가장 크게 짚어 둘 한계다. **제품을 영상에 온전하고 정확하게 담는 능력이 약하다.** 라벨,
용기 형태, 색, 로고 같은 실제 디테일이 컷에서 뭉개지거나 미묘하게 달라진다. 제품 홍보
숏폼에서 이건 사소한 흠이 아니라 핵심 기능의 약점이다.

원인은 생성 방식이다. 지금은 컷마다 제품 이미지를 start 프레임으로 넣지 않고 세그먼트를
멀티샷으로 한 번에 만든다. 한 세그먼트로 8초 이상을 만들기 어렵기 때문에 보통 숏폼에서 2개의 세그먼트를 생성후 잘라서 쓴다. 그래서 제품이 세그먼트별로 영상 모델의 내부 표현을 거치며 점점 원본과 멀어진다.
실제로 확실한 방안은, 제품 클로즈업은 제품 이미지를 강조한 상태로 별도로 생성해 편집 단계에서
멀티샷 본편과 이어 붙이는 것이다. 추후 과제로 둔다.

## 2. 핵심 한계: 나레이션과 목소리 다듬기 (중요, 미완)

**나레이션과 목소리(ElevenLabs 등)를 제대로 다듬지 못했다.** 숏폼에서 목소리는 제품 표현만큼
몰입을 좌우하는데 아직 거칠다.

- **캐릭터에 맞는 보이스 선택이 약하다.** 지금은 페르소나 성별만 읽어 프리메이드 목록의 첫
  보이스(여=Bella, 남=Adam)를 그대로 선택하는 경우가 많다. 나이대와 톤 매칭이 아니라 사실상 고정이라 서로
  다른 캐릭터가 같은 목소리로 나온다.
- **언어별 보이스 보장이 없다.** eleven_v3가 다국어여도 프리메이드마다 한국어 발음과 억양
  품질이 다른데, 지금은 언어를 선택에 반영하지 않는다.
- **믹스 밸런스와 연기가 거칠다.** BGM 덕킹과 발화 loudnorm의 균형, eleven_v3 오디오 태그
  매핑 모두 더 정교한 튜닝이 필요하다.

제품 충실도와 함께 다음 개선 1순위다.

## 3. 핵심 한계: reference-to-video 폐기

처음에는 Kling O3 reference-to-video가 캐릭터와 제품 일관성의 최선이라 봤지만, 실제 생성
퀄리티가 크게 못 미쳐(2026-07-02 확인) 운영 경로에서 버리고 기본을 Kling O3 Pro
image-to-video로 바꿨다. 코드는 남겨 뒀지만 기본으로는 선택되지 않는다(자세한 경위는
[docs/ToolnModels.md](docs/ToolnModels.md)).

## 4. 현재 구조의 가정과 가설

맞다고 증명된 게 아니라, 맞다고 보고 구조를 그쪽으로 기울인 가설이다.

- **스타일은 하드코딩이 아니라 측정 대상이다.** 레퍼런스에서 컷 리듬, 오디오 다이내믹,
  팔레트, 톤을 뽑아 `VideoProfile`로 만들고 생성은 그 데이터를 씨앗으로 돈다. 빠른 컷과 느린
  컷 레퍼런스가 서로 다른 리듬을 만들어야 "같은 엔진, 다른 입력, 다른 결과"가 성립한다.
- **분석기가 곧 채점기다 (단일 잣대).** 레퍼런스를 프로파일링하는 코드가 생성물도
  프로파일링한다. 둘이 같은 자로 재지고, 잘 만든 레퍼런스가 검증을 통과한다는 전제를 회귀
  기준선으로 썼다.
- **스키마가 유일한 경계다.** 분석과 생성은 pydantic 스키마로만 통신한다. 가장 바뀌기 쉬운
  이미지와 영상 백엔드를 스키마 뒤에 가두면 백엔드 교체가 한 단계만 건드린다.
- **결정론 층이 우선이고 모델은 그 위에 얹힌다.** 컷, 오디오, 팔레트 수치는 로컬 라이브러리가
  재현 가능하게 내고 모델은 지각적 묘사만 더한다. 결정론 수치를 모델이 덮어쓰지 않아, 키
  없이도 모델이 흔들려도 분석이 안정적으로 돈다.
- **로컬 단독 실행이면 충분하다.** ffmpeg 파이프라인이 서버리스 제약(타임아웃, 파일시스템)에
  안 맞아 웹 프런트와 서버리스 배포는 범위 밖이다. 코어는 CLI가 인프로세스로 부르는 패키지라
  나중에 웹 계층이 같은 코어를 감쌀 수 있게만 열어 뒀다.
- **단일 키로 진입 장벽을 낮춘다.** 필수 키는 `GEMINI_API_KEY` 하나로 분석과 이미지 생성을
  덮고, GCP 자격이 있으면 Vertex lane을 우선하되 단일 키를 폴백으로 보장한다.
- **voice는 나레이션을 기본으로 한다.** voice를 먼저 만들어 립싱크로 주입하는 방식은 무리였고,
  on-camera 립싱크는 컷이 나뉘면 톤이 흔들렸다. 그래서 화면과 분리된 나레이션으로 음색을 길게
  뽑고, on-camera는 꼭 필요할 때만 쓴다. 영상 모델 예산이 없으면 스틸을 켄 번스 모션으로 같은
  조립 경로에 태워 끝까지 돌린다.
- **사람 확인은 노드별 게이트가 아니라 chat의 확인과 수정 루프로 넣었다.** 모든 단계 뒤에 같은
  게이트 추상(ask/pass/run)을 배선하는 초안 대신, `chat`이 ReelProfile과 대표이미지를 보여주고
  수정 요청을 반영해 다시 만든다. 노드 단위 in-graph 게이트는 범위 밖이다(6절).
- **결정은 코드가 아니라 노드별 LLM이 문맥으로 내린다.** 캐릭터 외모, 음악, 후크, 촬영 환경,
  대사, 스토리보드 컷을 하드코딩하지 않고 각 노드가 입력과 레퍼런스, 주인공 문맥에 맞게
  정한다. 기본값은 '편향'일 뿐, 입력이 분명히 다른 걸 원하면 그걸 존중한다.
- **plan과 execute를 LangGraph로 표현한다(2026-07-01 전환).** 순차 함수였던 초안을
  `StateGraph`로 이관했다([workflows.md]). plan은 조건부 엣지로 hook과 storyboard 핑퐁을 돈다.
  storyboard가 훅을 전체 스토리에 녹여 보고 안 맞으면 개선 힌트와 함께 hook 노드를
  재호출한다(재시도 상한).

## 5. 모델 비교

용도별 선택과 근거는 README와 [docs/ToolnModels.md](docs/ToolnModels.md)에 정리했다. 회고로서
한 가지만 남긴다. **기획과 카피 LLM만 Gemini 3.1 Pro와 Claude Opus를 나란히 비교한다.** 두
키가 이미 있어 추가 비용 없이 같은 작업을 둘에 돌려 카피와 내러티브 품질을 보고, 잠정 1차는
Gemini 3.1 Pro로 두되 작업별로 나은 쪽을 굳혀 간다. 나머지 용도는 단일 키 안에서 도는 기본
경로를 우선했다.

## 6. 당장 포기했지만 아쉬운 것 (추후 후보)

시간과 범위 때문에 접었지만 여유가 생기면 붙일 것들이다.

### 파이프라인 그래프에서 미룬 것

지금 그래프는 게이트 없이 입력에서 ReelProfile을 거쳐 production까지 일괄로 돈다.

- **그래프 안 노드별 HITL 게이트.** 사람 확인은 `chat`으로 넣었고, 남은 것은 그래프 내부에서
  후크 후보를 고르고 중간 산출물(asset_bible, storyboard)을 확인하고 수정하는 세밀한 in-graph
  게이트다. 지금 plan과 run 경로는 후크 0번 후보를 자동 채택한다.
- **verify 하드 게이트와 repair 유한 루프.** 골격은 구현됐다. `verify`가 조건 엣지로 `assemble`
  또는 `describe`로 갈리고 교정 파라미터를 실어 최대 3회 되돌린다. 결정론 노드는 그냥 재실행하면
  같은 fail이 반복되므로 교정값을 주입하는 축부터 넣었고, 지금은 loudness 하나다. 약한 샷
  재생성이나 자막 위치 등 다른 축은 같은 틀에 향후 얹는다.
- **오디오 fan-out을 `Send` API로.** 지금은 `visuals` 뒤 `voice`, `bgm`, `sfx`를 정적 엣지
  병렬로 돌린다(fan-in은 `assemble`). 재료 수가 늘면 `Send`로 바꾼다.
- **canvas 샷 렌더러와 별도 환경 레퍼런스 이미지.** i2v와 켄 번스 두 경로로 좁혔다.
  `environment`는 텍스트 스펙만 만들고 `character` 뒤에 돌아 배경에 반영되지 않는다. 공간
  일관성이 중요해지면 환경 이미지를 별도 생성하고 environment를 character 앞으로 당긴다.

### 그밖에 접은 것

- **전용 토킹헤드 아바타.** HeyGen류 립싱크 아바타는 구현 부담과 깨짐 때문에 만들지 않는다.
- **기획과 카피 LLM 자동 라우팅.** 지금은 두 모델을 명시 우선순위로 비교만 한다. 작업 유형별
  자동 선택은 현 규모에 과해 접었다.
- **영상 모델 보류와 제외**(Seedance 검열 리스크, Kling과 Veo의 start image 컷 제어 노트)는
  [docs/ToolnModels.md](docs/ToolnModels.md)에 정리했다.
- **상위 이미지 모델 승격, whisperX 강제 정렬, 외부 스톡 음악, 관측 기반 튜닝 루프.** 지금
  경로에선 불필요해 미뤘다. 스틸 품질이 영상으로 전파되니 룩이 결과를 좌우하는 컨셉에서 상위
  이미지 모델을 다시 볼 만하고, 보이스오버를 본격적으로 켜면 whisperX 자막 정렬이 필요해진다.
  로컬 trace와 Langfuse는 배선했으나 점수와 비용을 모아 자동 개선하는 루프까지는 못 갔다.
- **재분할을 pacing에 맞춰 조절.** 영상은 세그먼트 1회 호출로 만들고 그 원본을 패널 경계로
  다시 잘라 컷마다 zoom과 크롭을 달리해 이어붙인다. 한 run(7.1초와 3.6초 세그먼트 2개)에서
  재분할 유무만 바꾸니 원본은 5컷 평균 2.15초, 재분할본은 8컷 평균 1.35초로 갈렸다. 대부분은
  켜는 게 맞지만 재인코딩 비용이 들고 느린 레퍼런스엔 과할 수 있어, `style.pacing`이 느리면
  약하게 쓰는 분기를 향후 넣는다.
- **replan에 변주 신호 주입.** replan을 돌렸는데 훅과 스토리가 이전과 거의 같았다. replan이
  원본 style을 통째 복사하고 narrative 노드만 다시 돌려 style이 동결되고, 훅 생성은 temperature
  0.9여도 brief가 같고 "다르게 가라"는 신호가 없어 같은 결과로 수렴한 탓이다. replan brief에
  이전 훅과 스토리를 피하라는 지시를 넣고 hook_type을 로테이션하면 갈라진다. `run_replan`
  시딩과 `_hook_node` brief 두 곳만 손대면 된다.
- **style을 핵심 노드로 승격(부분 반영) + 템플릿/인스턴스 분리(미래).** `StyleDimensions`는
  concept 노드가 채운다고 약속하지만 레퍼런스가 없으면 채우는 노드가 없어 빈 기본값으로 남았다.
  style은 hook과 story, music, execute를 모두 좌우하는 축이라, 이번에 그래프 구조는 두고 style을
  필수 노드로 만들어 hook과 story 앞에서 채우고 핑퐁 뒤 한 번 보정한다. 더 큰 방향은 레퍼런스가
  style과 storyboard를 담은 "템플릿"이 되고 plan은 그걸 특정 제품과 인물로 바인딩하는
  인스턴스화로 나누는 구조인데("같은 광고, 다른 제품"), 스키마(ReelTemplate)가 필요해 다음
  반복으로 미룬다.

## 7. analyze, verify, evaluate, compare 설계

네 명령은 "분석기가 곧 채점기"라는 한 원리에서 갈라진 단일 잣대다. 레퍼런스와 생성물에 같은
코드를 대고, 그래프가 없어도 단독 CLI로 동작한다. 순서가 있다. analyze가 프로파일
(`VideoProfile`)을 만들고, verify가 하드 pass/fail로 먼저 거른 뒤 통과한 것만 evaluate가 소프트
점수(0~100)를 내며, 레퍼런스가 있으면 compare가 두 프로파일을 대조한다. 사용법과 채점 기준은
README와 [../specs/rubric.md](specs/rubric.md)에 있고, 여기서는 설계 결정만 적는다.

- **analyze**: 결정론 로컬 층(ffprobe, PySceneDetect, librosa, OpenCV)이 재현 가능한 수치를
  내고 Gemini 층이 지각 라벨만 더한다. 결정론 수치를 모델이 덮어쓰지 않는 게 핵심이다.
- **verify (Conformance)**: 결과물이 의도한 템플릿대로 온전한지 하드로 검증한다
  ([../specs/conformance-gate.md](specs/conformance-gate.md)). execute 그래프에서는 하드 게이트로
  동작해 교정 가능한 fail이면 문제 노드로 되돌려 재생성하고, 잘 만든 레퍼런스는 모두 PASS여야
  한다(회귀 기준선).
- **compare (Similarity)**: 생성물이 레퍼런스와 같은 스타일인지 7축(rhythm, voice, music,
  visual, subtitle, tone, narrative)을 0~1로 채점해 신뢰도 가중으로 합산한다
  ([../specs/similarity-loop.md](specs/similarity-loop.md)). 정형 신호는 무겁게, Gemini
  자유텍스트 라벨은 어휘가 흔들려 가볍게 실었다. `run --max-iters>1`이면 미달 축의 델타를 plan
  피드백으로 밀어 넣어 재생성하고, 레퍼런스는 루프 전 한 번만 analyze해 재사용한다.

산출물은 분석은 `profiles/`, 검증과 평가는 `evals/`, 유사도는 생성물 폴더의 `similarity.json`에
남기며 모두 gitignore다.
