# 멀티샷 세그먼트 생성 (2026-07-01)

## 왜

기존 execute는 **컷마다 스틸을 만들고 컷마다 영상 모델을 호출**했다. 이는 두 가지로 틀렸다.

1. **멀티샷에 스틸 컷별 주입은 무의미하다.** 멀티샷 생성은 앵커 이미지 1장과 샷 리스트
   프롬프트를 받아 **모델이 내부에서 여러 컷을 만들어** 낸다. 컷마다 별도 이미지를 넣는
   구조가 아니므로, 컷 수만큼 스틸을 만들면 대부분 버려진다.
2. **호출 횟수가 과하다.** ≤15초 영상은 영상 모델 호출이 **최대 2회**여야 한다
   (참고 [trd.md](trd.md): 14초 = 7초 멀티샷 2개). 컷당 호출(9회)은 비용·시간 낭비다.

## 규칙 (binding)

- **세그먼트 = 영상 모델 1회 호출 단위.** 연속 패널을 모델 최대 클립 길이
  (`ModelCapability.max_clip_sec`, Veo 8초 / Kling 15초) 이하로 묶는다. 14초·Veo면 2개.
- **세그먼트당 앵커 이미지 1장만 생성·주입한다.** 세그먼트의 첫 패널 스틸이 앵커다. 나머지
  내부 컷은 멀티샷 프롬프트(샷 리스트)로 모델이 만든다. 컷마다 스틸을 만들지 않는다.
- **세그먼트 연결.** 2번째 이후 세그먼트는 **직전 세그먼트의 마지막 프레임**을 start
  image로 재사용해 캐릭터·톤 드리프트를 막는다.
- **멀티샷 프롬프트**는 세그먼트 안 패널들을 샷 리스트로 편다: 각 컷의 shot_type,
  피사체(제품 강조 컷이면 제품), beat 동작, 카메라 무빙(제품 컷은 제품으로 줌인)을 담는다.
- **ken_burns 폴백은 예외다.** 로컬 합성이라 영상 API 호출이 없다. 컷당 스틸 + 줌으로
  그대로 둔다(세그먼트 = 패널 1개). 워킹 스켈레톤 기본 경로를 지킨다.
- **자막은 생성이 아니라 편집 오버레이.** 패널별 자막 PNG를 최종 타임라인의
  `t_start~t_end` 구간에 시간 기반으로 덮는다. 세그먼트 2개든 컷 9개든 동일하게 동작한다.
- **편집단계 beat-cut 몽타주.** 세그먼트(연속 영상)를 패널 경계로 재분할해 컷마다 줌
  프레이밍을 크게 번갈아(제품 컷은 제품으로 더 세게 줌인) 인접 서브컷 경계를 다르게 한다.
  ≤2회 생성 호출을 유지하면서 fast_montage 컷 리듬(레퍼런스의 짧은 컷들)을 복원한다.
  입력이 실 영상이라 컷 내부는 움직여 정지가 아니다. ken_burns 폴백은 앵커 스틸 하나라
  재분할하지 않는다(그대로 한 컷).

## 인터페이스

- `ProductionPlan.segments: list[list[int]]` — 패널 인덱스 그룹. 호출 1회당 한 그룹.
  ken_burns면 `[[0],[1],...]`, 영상 모델이면 max_clip_sec로 묶은 그룹.
- `Materials.subtitle_spans: list[list[float]]` — 자막 PNG와 평행한 `[start, end]` 구간.
- `stills.ensure_panel_stills(..., anchor_indices)` — 앵커 패널만 스틸을 채운다
  (영상 모델 경로). None이면 전 패널(ken_burns).
- `materials`는 `plan.segments`를 돌며 세그먼트당 1클립을 만든다. 영상 백엔드면 멀티샷
  프롬프트 + 앵커/직전 마지막 프레임으로 1회 호출, ken_burns면 패널 스틸 줌.

## done-criteria

- ≤15초·Veo 실행 시 영상 모델 호출이 2회 이하다(로그·`.veo.mp4` 산출물 수로 확인).
- 세그먼트당 스틸 1장만 생성된다(앵커).
- ken_burns 경로 결과는 기존과 동일(컷당 줌, 자막 타이밍 유지).
- conformance 통과(길이·자막·볼륨), 자막이 패널 구간에 맞게 뜬다.
