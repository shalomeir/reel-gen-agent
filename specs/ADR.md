# 아키텍처 결정 기록 (ADR)

상태: 확정, 갱신형. 이 문서는 현재 에이전트 아키텍처가 어떤 결정 위에 서 있고, 각
결정을 왜 그렇게 내렸는지를 한곳에 모은다. 결정 하나가 한 레코드다. 결정을 바꾸면
해당 레코드의 상태를 갱신하고(예: Superseded), 새 레코드를 추가한다.

각 결정의 더 자세한 계약은 링크된 `specs/` 문서에, 배경 설명은 `docs/`에 있다. 숫자나
인터페이스가 어긋나면 링크된 정본 문서를 따른다. 회고(무엇을 써 보고 어땠는지)는
[../docs/Retrospective.md](../docs/Retrospective.md)에 따로 둔다.

## 레코드 형식

각 ADR은 **배경**(왜 이 결정이 필요했나), **결정**(무엇으로 정했나), **결과**(그래서
얻은 것과 감수하는 것)로 적는다. 상태는 Accepted(채택), Superseded(대체됨),
Deprecated(폐기) 중 하나다.

## ADR-0001: 로컬 단독 CLI, 서버/serverless 없음

**상태**: Accepted

**배경**: 영상 파이프라인은 ffmpeg에 기댄다. serverless에는 ffmpeg가 없고 타임아웃과
파일시스템 제약으로 영상 조립이 돌지 않는다. 별도 API 서버를 띄우면 운영 비용과
배포 복잡도가 붙는다.

**결정**: 노트북에서 도는 로컬 단독 CLI로 만든다. 웹 프런트엔드와 serverless 배포는
범위 밖이다. 별도 서버(FastAPI 등)를 두지 않고 CLI가 코어를 같은 프로세스에서 부른다.

**결과**: 한 줄 실행이 가능하고 채점/재현이 쉽다. 대신 동시 다중 사용자나 호스팅형
서비스는 이 결정으로는 못 한다. 코어를 패키지로 분리해 두어(ADR-0002) 나중에 웹 계층이
같은 코어를 감쌀 길만 열어 둔다.

## ADR-0002: LangGraph 코어를 CLI와 분리해 인프로세스로 호출

**상태**: Accepted

**배경**: 생성은 컨셉, 에셋 바이블, 스토리보드, 영상, 검증으로 이어지는 다단계 흐름이고
단계마다 사람 확인 지점이 있다. 이 흐름과 UI를 한 덩어리로 짜면 둘이 얽힌다.

**결정**: 코어를 LangGraph 그래프로 표현한다(노드 = 단계, interrupt = 게이트). CLI(typer +
rich)는 이 코어와 분리된 패키지로, 같은 프로세스에서 직접 호출한다.

**결과**: 단계와 게이트가 그래프로 일반화되고, UI 교체나 미래의 웹 계층이 코어를 건드리지
않는다. 대신 LangGraph 의존이 생긴다.

## ADR-0003: pydantic 스키마가 분석과 생성의 유일한 경계

**상태**: Accepted

**배경**: 이미지와 영상 백엔드는 가장 빠르게 바뀌는 부분이다. 분석 내부 구현에 생성
단계가 직접 묶이면 백엔드 교체가 시스템 전체로 번진다.

**결정**: 분석과 생성은 pydantic 스키마(`analysis/profile.py`, `generate/schema.py`)로만
통신한다. 스키마를 고정하면 백엔드를 갈아끼워도 한 단계만 손댄다.

**결과**: 백엔드 교체 비용이 한곳에 갇힌다. 경계를 흐리는 변경(생성 단계가 분석 내부에
직접 의존)은 거부하고 설계를 다시 본다. 계약: [architecture.md](../docs/architecture.md).

## ADR-0004: 두 층 분석, 결정론 수치 우선

**상태**: Accepted

**배경**: 모델 묘사는 호출마다 흔들리고 키가 없으면 못 돈다. 컷, 오디오, 팔레트 같은
수치는 재현 가능해야 회귀 기준선으로 쓸 수 있다.

**결정**: 분석을 두 층으로 짠다. 결정론 로컬 층(ffprobe, PySceneDetect, librosa, OpenCV)이
재현 가능한 수치를 내고, Gemini 층이 톤/자막/후크 같은 지각 필드를 더한다. 결정론 수치는
지각 층이 덮어쓰지 않는다.

**결과**: 분석이 키 없이도(`--no-gemini`) 돌고 수치가 안정적이다. 지각 묘사는 옵션 보강으로
남는다.

## ADR-0005: 분석기가 곧 채점기(단일 잣대)

**상태**: Accepted

**배경**: 생성물을 별도 기준으로 채점하면 레퍼런스와 생성물이 다른 자로 재진다. 그러면
"레퍼런스를 닮았나"를 일관되게 말할 수 없다.

**결정**: 레퍼런스를 프로파일링하는 코드가 생성물도 프로파일링한다. 같은 프로파일러가
스토리보드 씨앗 공급자이자 게이트 채점기다.

**결과**: 레퍼런스와 생성물이 한 자로 재진다. 잘 만든 레퍼런스가 검증을 통과한다는 전제를
회귀 기준선으로 쓴다([testing-strategy.md](testing-strategy.md)).

## ADR-0006: 검증 게이트 둘, conformance(하드) → rubric(소프트)

**상태**: Accepted

**배경**: "기술적으로 온전히 만들어졌나"와 "콘텐츠로서 먹히나"는 다른 질문이다. 깨진
결과물에 콘텐츠 점수를 매기는 건 의미가 없다.

**결정**: 영상 단계 뒤에 게이트를 둘 둔다. 먼저 conformance(하드 pass/fail, 무결성과
적합성), 통과하면 rubric(소프트 0~100, 콘텐츠 효과성). conformance가 fail이면 rubric으로
가지 않고 결함 카테고리로 해당 샷 재생성을 트리거한다. 두 게이트는 그래프 노드인 동시에
단독 CLI(`verify`, `evaluate`)로도 돈다.

**결과**: 무결성과 효과성을 분리해 판정하고, 그래프 없이도 레퍼런스에 지금 바로 적용
가능하다. 계약: [conformance-gate.md](conformance-gate.md), [rubric.md](rubric.md).

## ADR-0007: 사람 확인 게이트 일반화(ask/pass/run)

**상태**: Accepted

**배경**: 대화형으로 단계마다 확인하고 싶을 때도, 한 번에 끝까지 돌리고 싶을 때도 있다.
모드마다 따로 코드를 짜면 챗과 런이 어긋난다.

**결정**: 모든 중요한 단계 뒤에 같은 게이트 추상을 둔다. ask(챗 기본, 확인/수정),
pass(`--force-step-pass <step>`), 런 모드(전부 자동 통과)가 한 그래프 위에서 동작한다.

**결과**: 챗 모드와 런 모드가 같은 그래프를 공유한다. 새 단계를 더할 때 게이트를 같은
방식으로 배선하면 두 모드가 자동으로 일관된다. 계약: [product-design.md](product-design.md).

## ADR-0008: 모델 비종속, ID는 .env로 주입(어댑터 뒤에 둠)

**상태**: Accepted

**배경**: 모델은 자주 바뀌고 셧다운되기도 한다. 모델 ID를 코드에 박으면 교체가 코드
수정이 된다.

**결정**: 모델 ID와 백엔드는 `.env`로 주입하고 SDK 어댑터가 읽는다. 코드와 TRD는 특정
모델에 묶이지 않는다. 어떤 모델을 왜 골랐는지는
[ai-model-records.md](ai-model-records.md)에 따로 기록한다.

**결과**: 모델 교체가 어댑터 한 곳과 `.env` 수정으로 끝난다. 선택 근거와 코드가 분리돼
이력 관리가 쉽다.

## ADR-0009: 단일 필수 키와 Vertex 우선 폴백

**상태**: Accepted

**배경**: 채점자나 새 사용자가 키 여러 개를 꽂아야 돌아가면 진입 장벽이 높다. 한편 GCP
크레딧이 있으면 그걸 우선 쓰는 게 이득이다.

**결정**: 필수 키는 `GEMINI_API_KEY` 하나로 분석과 이미지 생성을 덮는다. 멀티모달 호출은
`GENAI_BACKEND=auto`로, GCP 자격이 있으면 Vertex lane을 우선해 크레딧을 쓰고 없으면 단일
키 lane으로 내려간다. 단일 키 약속은 폴백으로 항상 보장한다.

**결과**: 키 하나만으로 분석이 돈다. GCP 크레딧이 있는 환경은 자동으로 그쪽을 탄다.
예외는 기획·카피 텍스트 레인으로, 품질 비교를 위해 Gemini와 Claude 두 키를 일급으로
쓴다(ADR-0008의 기록 문서 2번 참고).

## ADR-0010: Veo 3.1은 Vertex lane 전용, fal.ai는 별도 용도

**상태**: Accepted

**배경**: 같은 영상 모델을 여러 lane으로 호출하면 출력 위치와 과금이 갈려 운영이
복잡해진다. 영상은 가장 비싼 단계라 경로를 단순하게 못박을 필요가 있다.

**결정**: 기본 영상 모델은 Veo 3.1 Fast(`veo-3.1-fast-generate-001`)로 고정하고 항상
Vertex AI lane으로만 호출한다(출력은 GCS, `VEO_OUTPUT_GCS_URI`로 통일). 중요한 히어로 컷은
Veo 3.1 Standard/Pro(`veo-3.1-generate-001`)로만 승격한다. Veo 3.1 Lite는 start/end control
board에서 split artifact가 생겨 기본 경로에서 제외한다. Gemini API Veo lane은 쓰지 않는다.
fal.ai는 사용자가 공식 요청할 때의 Kling O3/Seedance 영상 비교에만 쓴다(이미지는 Nano Banana
단일 경로). Veo를 fal로 호출하지 않는다.

**결과**: 영상 경로가 단순해지고 크레딧 사용이 예측 가능하다. 컷 제어는 모델을 늘리기보다
Nano Banana start image와 프롬프트로 해결한다. Kling O3/Seedance는 별도 비교 요청이 있을
때만 같은 인터페이스 뒤에 붙인다. 계약:
[ai-model-records.md](ai-model-records.md) 4번.

## ADR-0011: 저비용 켄 번스 폴백을 기본 경로로

**상태**: Accepted

**배경**: 영상 모델은 비싸고, 예산이 없으면 파이프라인이 끝까지 안 돌면 시스템을 증명할
수 없다.

**결정**: 영상 백엔드를 끄면 스틸을 켄 번스 모션으로 같은 조립 경로(ffmpeg)에 태운다.
이게 워킹 스켈레톤의 기본 경로다.

**결과**: 영상 모델 예산 없이도 완성된 mp4가 끝까지 나온다. 영상 모델은 품질 업그레이드로
얹는다.

## ADR-0012: voice를 되도록 사용, 립싱크는 영상 모델 네이티브만

**상태**: Accepted(개정). 초안의 "주 경로는 자막+뮤직 베드, voice 기본 꺼짐, 립싱크 제외"를
아래로 개정한다.

**배경**: voice는 숏폼 설득력과 전환에 크게 기여한다. 실측 결과, 화면 인물이 직접 말하는
on-camera 립싱크는 구현이 까다롭고(특히 컷이 나뉘면 컷마다 목소리 톤이 달라진다), voice를
먼저 생성해 영상 모델에 주입해 립싱크를 맞추는 방식은 무리였다. 반면 화면과 분리된 나레이션
(TTS)은 캐릭터 개성을 살린 목소리를 길게 연속으로 뽑을 수 있어 자연스럽고 구현이 쉽다.

**결정**: **voice는 되도록 사용하되, 기본 전달은 나레이션(`voiceover`)이다.** 전달 방식은
셋이다. (1) `voiceover` - 화면과 분리된 TTS 나레이션. **기본 경로.** ElevenLabs(선호) 또는
Google TTS Chirp 3로 캐릭터 음색을 길게 뽑아 컷이 나뉘어도 톤이 일관된다. (2) `on_camera` -
영상 모델이 네이티브로 등장인물을 말하게 함. 역동적으로 화면 인물이 직접 말해야 할 때만.
원컷이면 Veo/Kling 둘 다 가능하지만 **여러 컷에서 목소리를 일관 유지하며 립싱크해야 하면
`Kling O3 Pro reference-to-video`가 유일**하다. (3) `none` - 음악 베드만. **voice를 먼저
생성해 영상 모델에 주입하는 립싱크는 시도하지 않는다.** 전용 립싱크 아바타 파이프라인
(HeyGen류)도 만들지 않는다. voice는 어느 경로든 캐릭터 설정(`ModelSpec`)에서 음색을 유도한다.

**결과**: 립싱크에 욕심내지 않고 나레이션으로 자연스럽게 간다. on_camera는 꼭 필요할 때만
쓰고, 멀티컷 일관성이 필요하면 Kling O3 Pro로 올린다. 실제 전략은 ProductionPlan이 해소하고
폴백을 기록한다. 캐릭터 매칭 voice 생성과 정렬이 핵심 요건이다([information-schema.md](information-schema.md)
NarrationSpec, [workflows.md](workflows.md) scripting/assemble).

## ADR-0013: 로컬 trace가 진실의 원천, Langfuse는 옵션 sink

**상태**: Accepted

**배경**: 관측을 특정 백엔드에 묶으면 키가 없거나 그 서비스가 죽었을 때 실행 기록이
사라진다. 동시에 트레이싱 대시보드의 이점도 쓰고 싶다.

**결정**: 그래프는 `TraceEvent`를 emit하고 여러 sink가 구독한다. 로컬 JSONL sink는 항상
켜진 진실의 원천이고(`logs/<session_id>/<run_id>/trace.jsonl`), Langfuse sink는
`LANGFUSE_*`가 있을 때만 붙는다. Langfuse 실패는 실행을 막지 않는다.

**결과**: 키 없이도 실행이 온전히 남고, 키가 있으면 Langfuse 대시보드를 얻는다. 계약:
[logging-strategy.md](logging-strategy.md).

## ADR-0014: uv와 프로젝트 로컬 .venv로만 실행

**상태**: Accepted

**배경**: 글로벌이나 pyenv 파이썬으로 돌리면 의존성이 환경마다 달라져 재현이 깨진다.

**결정**: 의존성은 uv로 관리하고 항상 프로젝트 로컬 `.venv`에서 실행한다. 글로벌/pyenv
파이썬에 설치하거나 거기서 실행하지 않는다.

**결과**: 환경이 재현 가능해진다. 비대화형 셸은 direnv 훅이 안 걸리니 `.venv`를 명시
활성화하거나 venv 바이너리를 직접 부른다.

## ADR-0015: specs가 구속력, docs는 자료

**상태**: Accepted

**배경**: 설계 문서가 늘면 무엇이 구현을 강제하는 계약이고 무엇이 배경 설명인지 흐려진다.

**결정**: `specs/`만 구현을 강제하는 정본이다. `docs/`는 배경과 근거를 담는 자료지 계약이
아니다. 코드와 `specs/`가 어긋나면 spec이 이긴다(또는 spec이 틀렸으면 spec부터 고친다).
`docs/` 설계가 채택되면 구속력 있는 부분을 `specs/`로 승격한다.

**결과**: 무엇을 따라야 하는지가 분명해진다. 두 곳에 중복으로 쓰지 않고 링크로 잇는다.
